# [WORKFLOW]: MERGE CONFLICT RESOLUTION
> **ATTENTION MERGE AGENT:** You are tasked with resolving Git merge conflicts generated by concurrent multi-agent development. You must specifically focus on root lockfiles that Git cannot auto-merge.

## THE LOCKFILE RESOLUTION PROTOCOL (PNPM)
When multiple agents work concurrently on `apps/web-client` and `apps/backend-api`, they will both modify the root `pnpm-lock.yaml`.

When instructed to merge a Pull Request that has conflicts in `pnpm-lock.yaml`, you must strictly execute this sequence:

1. **Checkout Branch:** Check out the incoming Pull Request branch (e.g., `git checkout feature/frontend`).
2. **Merge Main:** Merge `main` into the branch (e.g., `git merge main`). Git will flag `pnpm-lock.yaml` as conflicted.
3. **Delete Lockfile:** Delete the corrupted lockfile completely: `rm pnpm-lock.yaml`.
4. **Regenerate Lockfile:** Run `pnpm install --no-frozen-lockfile`. PNPM will analyze all `package.json` files in the monorepo and generate a fresh, conflict-free lockfile combining dependencies from both `main` and the PR branch.
5. **Verify Build:** Run `turbo run build` to ensure the newly generated lockfile did not break the monorepo build pipeline.
6. **Commit & Push:** Stage the new lockfile and the resolved files: `git add .`, `git commit -m "Merge main and resolve lockfile conflicts"`, `git push`.

**STOP.** The branch is now clean and ready for the user to merge on GitHub.
